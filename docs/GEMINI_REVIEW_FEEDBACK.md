## Gemini 3.1 Pro 反馈

### 问题
> 当前 DEFAULT_REQUEST_TIMEOUT=300 秒。gemini-3.1-pro-preview 复杂任务需要 60-120 秒。是否需要增加到 600 秒？

### 回答
> **否。** 当前 300 秒的超时限制远高于 gemini-3.1-pro-preview 复杂任务所需的 60-120 秒，提供了充足的安全缓冲。除非单次请求需要连续多次调用该模型且总耗时超过 5 分钟，否则没有必要将其增加到 600 秒。保持 300 秒还能更快地发现和处理真正的服务挂起异常。

## 建议调整

根据 Gemini 的反馈，建议**保持 300 秒超时**，原因：

1. **充足的安全边际**: 300 秒是最大需求（120 秒）的 2.5 倍
2. **快速失败**: 能更快发现真正的服务挂起
3. **避免过长等待**: 600 秒可能导致用户等待过久

## 最终配置建议

| 配置项 | 建议值 | 理由 |
|--------|--------|------|
| `DEFAULT_REQUEST_TIMEOUT` | 300 秒 | 充足的安全边际 |
| `STARTUP_TIMEOUT` | 60 秒 | daemon 启动需要时间 |
| `HEALTH_CHECK_TIMEOUT` | 5 秒 | 避免网络波动误判 |

## 超时时间对比

```
gemini-3.1-pro-preview 响应时间测试:
├─ 简单任务：~20 秒
├─ 中等任务：~40 秒
├─ 复杂任务：~60-72 秒（实测）
└─ 最大需求：~120 秒（预估）

当前超时配置：300 秒
安全边际：300 / 120 = 2.5x ✅
```

## 使用建议

### 默认使用 300 秒
```python
acp_dispatch({
    "task": "复杂任务",
    "worker": "gemini-3-pro"
    # timeout 默认 300 秒
})
```

### 特殊场景延长超时
```python
# 仅在以下场景使用更长超时：
# 1. 批量任务（多次 API 调用）
# 2. 超大型代码库分析
# 3. 网络条件较差的环境
acp_dispatch({
    "task": "超复杂任务",
    "worker": "gemini-3-pro",
    "timeout": 600  # 手动指定
})
```

## 结论

✅ **推荐配置：300 秒默认超时**
- 对于 99% 的场景足够
- 特殊场景可手动延长
- 平衡了可靠性和响应速度
